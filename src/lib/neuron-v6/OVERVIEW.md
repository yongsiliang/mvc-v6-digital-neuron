## 🏗️ 系统架构总览

### 7层架构

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer 7: Application Layer                                      │
│   - Tauri桌面应用 / Web界面 / API接口                           │
├─────────────────────────────────────────────────────────────────┤
│ Layer 6: Autonomous Layer (自主层)                              │
│   - AutonomousCore: ReAct推理、任务编排、工具调用                │
│   - AutonomousLoop: 感知-思考-行动循环                          │
├─────────────────────────────────────────────────────────────────┤
│ Layer 5: Consciousness Core (意识核心)                          │
│   - 事件驱动架构、处理器路由、状态机管理                         │
│   - ThinkingHandler / LearningHandler / ProactiveHandler        │
├─────────────────────────────────────────────────────────────────┤
│ Layer 4: Meta-Cognition Layer (元认知层)                        │
│   - 元学习引擎、维度理解、算法反思                               │
│   - ★ 待实现：SSM状态空间模型 + MCTS规划                        │
├─────────────────────────────────────────────────────────────────┤
│ Layer 3: Memory Layer (记忆层)                                  │
│   - 工作记忆、长期记忆、动态上下文                               │
│   - 记忆压缩、洞见提取、关键信息抽取                             │
├─────────────────────────────────────────────────────────────────┤
│ Layer 2: Compiler Layer (编译层) - ⚠️ 过度设计                  │
│   - AttentionNetwork: 本地向量网络                              │
│   - ConsciousnessCompiler: 编译流程                             │
├─────────────────────────────────────────────────────────────────┤
│ Layer 1: Infrastructure Layer (基础设施)                        │
│   - LLM Gateway、Embedding Service、Vector DB                   │
│   - 配置管理、日志监控                                          │
└─────────────────────────────────────────────────────────────────┘
```

### 核心数据流

```
用户输入 
  → ConsciousnessCore.process() 
  → 事件路由到对应Handler 
  → Handler调用各子系统 
  → 结果返回用户
```

## 🔧 SSM融入计划

### 核心问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 状态分散 | 每个Handler独立维护状态 | 无法跨层传递，记忆碎片化 |
| 长期依赖弱 | 无系统级长期状态管理 | 上下文丢失，重复计算 |
| Token消耗高 | 所有语义处理都调LLM | 成本高，延迟大 |

### 推荐方案：SSM + MCTS 混合架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    元思考层重构                                  │
│                                                                 │
│   ┌───────────────┐      ┌───────────────┐                     │
│   │  SSM Encoder  │ ──→  │ SSM State     │ (统一状态空间)       │
│   │  (输入编码)    │      │ (历史压缩)    │                      │
│   └───────────────┘      └───────┬───────┘                     │
│                                  ↓                              │
│                         ┌───────────────┐                      │
│                         │ MCTS Planner  │                      │
│                         │ (决策规划)    │                      │
│                         └───────┬───────┘                      │
│                    ┌────────────┼────────────┐                 │
│                    ↓            ↓            ↓                 │
│              本地输出      需要LLM      需要工具                │
│                            ↓            ↓                     │
│                       LLM调用    工具调用                       │
└─────────────────────────────────────────────────────────────────┘
```

### 实现阶段

| 阶段 | 内容 | 核心文件 |
|------|------|----------|
| 阶段1 | 基础SSM实现 | `ssm-layer.ts`, `ssm-encoder.ts`, `ssm-decoder.ts` |
| 阶段2 | SSM+MCTS集成 | `ssm-mcts-controller.ts` |

### 预期收益

- **Token节省**: 60-80%（简单问题本地解决）
- **延迟降低**: 50%+（减少LLM调用）
- **上下文长度**: 无限（SSM压缩历史）

## 🤖 LLM定位与改造

### 当前问题

```
发现问题（150+处LLM调用）：

❌ 调用路径不统一（Gateway/Client混用）
❌ Token预算分散（每个模块独立计算）
❌ 没有全局监控（无法追踪总消耗）
❌ 缓存失效（多入口独立缓存）
```

### LLM理想位置

```
核心原则：
1. LLM是"执行器"，不是"决策者"
2. 元思考层决定"如何使用LLM"
3. 统一入口：所有LLM调用必须经过元思考层

决策流程：
输入 → 元思考层规划 → [LLM指令] → LLM Gateway执行 → 输出
                ↑
                └── 包含Token预算、提示词、期望输出等
```

### 改造方向

| 改造点 | 方案 |
|--------|------|
| 调用入口 | 所有模块通过MetaThinkingIntegrator请求LLM |
| Token预算 | 元思考层硬性控制，Gateway强制执行 |
| 辅助模块 | 优先使用本地算法，只在必要时调用LLM |
| 统计监控 | Gateway统一收集Token消耗、缓存命中等指标 |

3. **性能验证**
   - 每个阶段完成后进行性能测试
   - 对比改造前后的Token消耗和延迟
